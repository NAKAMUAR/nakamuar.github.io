# 音声操作型 Genspark エージェント 設計書（原文＋レビュー）

本書は当初設計の原文に、調査・レビュー結果の注釈を加えたものです。
実装は本レビューの結論を反映し、`README.md` の方針で進めています。

---

## 0. レビュー要約（最重要・後から追記）

実装着手前に設計書 §9 の指示どおり §7-第二（規約確認・公式機能の有無）を調査した結果、
**当初の「browser-use による Genspark 自動操作」方式は採用しないことを決定**した。

### 結論
1. **規約上の懸念。** Genspark の利用規約は「スクレイピング・データマイニング・抽出を含む
   体系的／自動的なデータ収集」および「本サービスを他サービス等の入力として利用すること」を
   禁止している。browser-use による自動操作はこれに該当し得る。
2. **公式機能が既に存在。** 音声入力は公式アプリ **Speakly**、自律操作・ワークフロー実行は
   公式 **Super Agent** が提供済み。§7-第二自身が「公式に同等機能があれば乗り換えを強く推奨」と
   述べており、本実装はそれに従う。
3. **公開 API は未提供（2026 時点）。** 完全プログラム連携は現状不可。

### 採用方針
ローカル音声認識（faster-whisper）＋ Gemini による「依頼文生成」までを自作し、
最後は**ユーザーの手動貼り付け／公式 Speakly** で Genspark に渡す**規約順守**の構成とする。

### その他のレビュー指摘
- **§3 のコスト「無料」は注意。** Gemini 無料枠はレート制限と「学習利用される可能性」が伴う。
  機密文章を扱う場合は有料枠/データ取り扱い条件の確認を推奨。
- **§4.1 compute_type。** CPU 運用では faster-whisper を `int8` にすると軽量・実用的。
- **§7-第三のレート制限。** ブラウザ自動操作を廃したことで 1 タスクあたりの API 呼び出しは
  「指示解釈の 1 回」に激減し、無料枠超過リスクは大幅に低下する（副次的メリット）。
- **§4.2 モデル名。** Flash-Lite 系のモデル ID は変動するため `.env` で差し替え可能にした。

## 1. 目的と全体像
ユーザーが音声で指示を出すと、その指示を解釈して Web サービス「Genspark」をブラウザ上で
自動操作し、文章校閲や画像の質感向上を実行するエージェントを構築する。重い処理は外部に
外出しし、手元のマシンとローカル計算を最小化する方針とする。

処理の流れは、音声入力 → 音声認識(テキスト化) → 指示解釈 → ブラウザ操作の判断と実行 →
結果取得、というパイプラインになる。

> 📝 注: 「ブラウザ操作の判断と実行」は §0 の結論により、本実装では
> 「依頼文生成 → 確認 → 手動/公式機能での受け渡し」に置き換えている。

## 2. 設計方針（最重要）
計算の重い部分はすべてクラウドに外出しする。文章校閲・画像加工は Genspark 側のサーバーで
実行され、操作判断は Gemini API のサーバーで実行される。そのため手元の PC は音声録音・
音声認識・ブラウザ操作・API 通信という軽量タスクのみを担い、高性能 GPU は不要、一般的な
ノート PC（メモリ 16GB、GPU 不要）で動作する想定とする。

## 3. コンポーネント構成
| 役割 | 採用技術 | 実行場所 | コスト |
|---|---|---|---|
| 音声認識(STT) | faster-whisper (small モデル) | ローカル | 無料 |
| 操作判断(エージェントの頭脳) | Gemini API (Flash-Lite 系) | クラウド | 無料枠 |
| ブラウザ操作 | ~~browser-use (OSS)~~ → **不採用**（§0） | ローカル | 無料 |
| 文章校閲・画像加工 | Genspark | クラウド | 有料(月額) |

## 4. 各コンポーネント詳細

### 4.1 音声認識（STT）
faster-whisper を使い、マイク入力をテキスト化する。モデルサイズは small を基準とし、精度が
不足する場合のみ medium へ。GPU なしの CPU 動作を前提とする。日本語認識に対応。
入力: マイクからの音声ストリーム。出力: 文字列の指示テキスト。
代替案として、ローカル負荷をさらに下げたい場合はクラウドの音声認識サービス（無料枠のある
もの）に差し替え可能な構造にしておく（→ `stt.py` の `SpeechToText` 抽象で実現）。

### 4.2 操作判断（エージェントの頭脳）
Gemini API を使用。モデルは Flash-Lite 系を基準とし、無料枠のレート制限内での運用を前提と
する。高精度が必要な判断のみ Flash 系へ切り替え可能にする（→ `.env` の `GEMINI_MODEL`）。
役割は、STT が出力した指示テキストを受け取り、Genspark 向けの最適な依頼文を生成すること
（当初は「次のブラウザ操作の決定」だったが §0 により役割を変更）。

### 4.3 ブラウザ操作 → 受け渡し（変更）
~~browser-use を使用。~~ §0 の結論により**自動操作は行わない**。代わりに、生成した依頼文を
確認のうえクリップボードへコピーし、ユーザーが Genspark／公式 Speakly に貼り付ける。
公式の自動化が必要なら Super Agent を利用する。

### 4.4 文章校閲・画像加工（Genspark）
Genspark 上で実行。実際の校閲・画像生成の計算は Genspark 側で行われ、ローカルには負荷が
かからない。

## 5. ハードウェア要件
メモリ 16GB（8GB でも動作可だが 16GB 推奨）、現行世代の一般的な CPU、GPU 不要、認識精度を
確保するためのまともな品質のマイク。直近 2〜3 年以内のミドルクラスのノート PC で要件を満たす。

## 6. コスト構造
ランニングコストは Genspark の月額のみ。音声認識・指示解釈はすべて無料（Gemini 無料枠内）。
ブラウザ自動操作を廃したため API 呼び出し回数が激減し、無料枠で安定運用しやすい。

## 7. 既知のリスクと考慮事項
- **第一（ブラウザ操作の脆さ）**: 自動操作を廃したため本リスクは解消。
- **第二（規約・公式機能）**: §0 で解決済み。公式機能へ寄せる方針を採用。
- **第三（無料枠レート制限）**: 指示解釈 1 回/タスクに削減され、リスク低下。判断エンジンは
  差し替え可能（`brain.py` の `Brain` 抽象）。
- **第四（自動実行の安全性）**: 受け渡し前に必ず確認ステップを挟む（`handoff.py`）。自動送信
  は行わない。

## 8. 将来の拡張オプション
判断もオフライン完結させたい場合は `Brain` 実装をローカル LLM に差し替える。推奨環境は
メモリ 32GB の Mac mini（Apple Silicon で 14B〜32B クラス）、または VRAM16GB 以上の NVIDIA
GPU 機。CUDA が導入容易で、AMD の ROCm はハードルが高い。70B クラスは本用途には過剰。

## 9. 推奨される進め方
最小構成（音声認識 ＋ Gemini 無料枠 ＋ 規約順守の受け渡し）をまず動かし、不満が出た部分のみ
強化する段階的アプローチ。§7-第二は実装前に解決済み。
