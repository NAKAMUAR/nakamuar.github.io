# 実装上の判断記録（DECISIONS）

指示書（§0）の方針に従い、曖昧な点は実装を止めず合理的なデフォルトを選び、ここに記録する。

## 1. アプリの配置場所：`memo-app/` サブディレクトリ

このリポジトリ（`nakamuar.github.io`）は既存の静的ポータル（`index.html`）を持つ GitHub Pages リポジトリ。
ルートに Next.js アプリを展開すると既存サイトを破壊するため、アプリは `memo-app/` 配下に作成した。
指示書 §7 のディレクトリ構成は `memo-app/` をプロジェクトルートとして対応している。

## 2. データ操作の責務分担（Server / Client）

- **認証ガードと初期データ取得**はサーバー（`@supabase/ssr` のサーバークライアント + `middleware.ts`）で実施。
  未ログイン時は middleware が `/login` へリダイレクト（§2.1, §5）。
- **メモの作成・アーカイブ・削除**は Server Actions（`app/actions.ts`）で実装。
- **メモ本文の編集**と**ファイル添付・削除・署名付き URL 取得**はクライアントのブラウザクライアントで実施。
  - ファイルアップロードは指示書 §2.3 / §6 の要求どおりクライアントから Storage へ直接行う（サーバー経由しない）。
  - いずれの操作も anon key + RLS で保護されるため安全。

## 3. メモ本文の保存方式：自動保存（デバウンス 800ms）

「保存」ボタン方式ではなく、入力停止後 800ms で自動保存する方式を採用。
離脱時（アンマウント時）にも未保存分をフラッシュする。理由：メモアプリとしての体験を優先。
保存状態は「保存中／保存しました／失敗」を画面に表示する。

## 4. 検索の実装：`ilike` による部分一致（OR 検索）

`title` と `body` に対し `%キーワード%` の `ilike` を `or` で結合した部分一致検索（§2.2 の「簡易キーワード検索」）。
全文検索（tsvector）は無料枠でも可能だが、要件は「簡易」であり、日本語の分かち書き設定が別途必要になるため採用しない。

## 5. アーカイブ表示の切り替え：URL クエリパラメータ

一覧は既定で `is_archived = false` のみ表示。`?archived=1` でアーカイブ済みのみ表示する。
検索クエリ `?q=` と併用可能。サーバーコンポーネントで再取得するため状態管理がシンプル。

## 6. Storage オブジェクトキーのサニタイズ

パスは指示書どおり `{user_id}/{note_id}/{uuid}-{filename}`。
ただしファイル名に日本語や記号が含まれると Storage キーとして問題が出るため、
キー部分のファイル名のみ `[^A-Za-z0-9._-]` を `_` に置換する。
表示用の元のファイル名は `attachments.file_name` にそのまま保存する（UUID 接頭辞で一意性は担保）。

## 7. 署名付き URL の有効期限：1 時間

`createSignedUrl(s)` の TTL を 3600 秒に設定（`lib/constants.ts`）。
一覧表示のたびにまとめて再発行する（`createSignedUrls`）。公開 URL は一切使用しない（§4.2）。

## 8. メモ削除時の添付クリーンアップ

DB は `attachments.note_id` の `on delete cascade` で行が消えるが、Storage の実体は自動削除されない。
そのため `deleteNote`（Server Action）で対象メモの全 `file_path` を先に Storage から削除してからメモ行を削除する。

## 9. 個別添付削除の順序

指示書 §2.3「Storage の実体と DB レコードの両方を削除」に従い、
Storage オブジェクト → DB レコードの順で削除する（先に実体を消し、孤児レコードを残さない）。
アップロード時に DB 挿入が失敗した場合は、アップロード済みオブジェクトをロールバック削除する。

## 10. カレンダー機能（Apple 標準カレンダー相当）

指示書に追加要望された「カレンダー（スケジュール機能）」を実装。「Apple の標準カレンダーと同じで良い」との指定に基づき、次の範囲を採用した。

- **ビュー**: iPhone カレンダーの基本である**月表示**を採用。1か月 = 6週 × 7日（日曜始まり）の固定グリッド。
  週/日ビューは Apple にもあるが、無料枠アプリの主目的（予定の把握・登録）には月表示で十分なため、初期実装では月表示のみとした。
- **予定の属性**: タイトル / 場所 / 終日フラグ / 開始・終了日時 / カラー / メモ（Apple の標準イベント編集画面に対応する主要項目）。
  参加者・通知・繰り返しは無料枠 + サーバーレス構成での通知基盤が必要になるため、初期実装では見送る（将来拡張余地として記載）。
- **カラー**: Apple のカレンダーカラーに倣い 6 色のプリセット（red/orange/yellow/green/blue/purple）。
  Tailwind の purge を考慮し、`lib/constants.ts` にクラス名を静的に列挙している。
- **終日イベントの保存**: `starts_at` = 当日 00:00（ローカル）、`ends_at` = 終了日 23:59（ローカル、最終日を含む）として timestamptz に保存。
  通常イベントは選択した開始・終了時刻をそのまま保存し、終了 < 開始の場合は終了を開始に合わせる。
- **タイムゾーン**: グリッドの日付計算はブラウザのローカルタイムで行い、DB へは UTC ISO 文字列で保存（`timestamptz`）。
- **データ取得**: 表示中の月グリッドの範囲（先頭セル〜末尾セル）に重なる予定だけを毎月クライアントから取得する
  （`starts_at < rangeEnd AND ends_at >= rangeStart`）。RLS により本人の予定のみ返る。
- **CRUD の責務**: メモと同様、予定の作成・編集・削除はクライアントのブラウザクライアント + RLS で実施（`EventEditor`）。
- **ナビゲーション**: メモ / カレンダーを切り替える共通ヘッダー `components/Nav.tsx` を追加。

DB は `0003_calendar.sql` に `events` テーブル・インデックス・RLS・`updated_at` トリガー（`set_updated_at()` を再利用）を定義。

## 11. Storage ポリシーをマイグレーションに同梱

バケット作成（private）と RLS ポリシーを `0002_storage.sql` に含めた。
これにより README の手順（SQL を貼って Run）だけで再現可能（§8, §9-6）。
バケット名・パス先頭セグメント（`storage.foldername(name)[1]`）と `auth.uid()` の一致でアクセス制御する。
